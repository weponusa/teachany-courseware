Core Concept

递归不是“重复”，而是“同类问题嵌套”

如果一个问题可以写成“先解决一个更小的同类问题，再补上当前这一层的动作”，就有递归结构。

def factorial(n): if n == 1: # 出口 return 1 return n * factorial(n - 1) # 规模变小

Divide and Conquer

分治三步：拆开，解决，合并

Divide

把原问题拆成规模更小、结构相同或相近的子问题。

Conquer

对子问题继续求解；小到基本情况时直接给答案。

Combine

把子问题答案合并成原问题答案。归并排序的关键就在这里。

递归强调“自我调用”；分治强调“拆分与合并”。两者经常一起出现，但不是同一个概念。

Merge Sort

归并排序：为什么需要“合并”这一步？

拆到单个元素很容易，难点是把两个有序小数组合并成一个更大的有序数组。

Recursion vs Iteration

递归和循环：谁更好？

维度递归循环
表达贴近树、分治、数学定义贴近连续步骤和状态更新
空间占调用栈通常更节省栈空间
风险出口错误会爆栈条件错误会死循环

判断原则

如果问题像树一样分叉、像俄罗斯套娃一样嵌套，递归通常清晰。如果只是从 1 加到 n，循环可能更直接。

高中阶段更重要的是能读懂递归过程，而不是盲目追求“递归写法”。
Worked Example

例题：用递归求列表最大值

思路：一个列表的最大值，可以看成“第一个元素”和“剩余列表最大值”的较大者。

def max_rec(a): if len(a) == 1: return a[0] rest = max_rec(a[1:]) return max(a[0], rest)

三要素标注

出口：len(a)==1

同类问题：max_rec(a[1:])

规模收缩：列表长度减少 1

合并：max(a[0], rest)

给出函数框架，学生补出口。

Concept Test

概念检测：哪个是合格的递归出口？

递归处理自然数 n，目标是从 n 一直处理到 0。哪个出口最稳？

检查递归出口时，总问自己：所有合法输入最终都能走到这里吗？
Complexity Intuition

复杂度直觉：为什么“每次一半”很快？

线性查找

n → n-1 → n-2 → ...

最坏可能看 n 次。

二分查找

n → n/2 → n/4 → n/8 ...

规模每次减半，约 log₂ n 层。

归并排序

拆分 log n 层，每层合并 n 个元素

总量约 n log n。
